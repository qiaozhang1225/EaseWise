# Stability Output Contract

## Required Fields

稳定性最终结构化输出保留以下字段：

- `verdict`
- `content`
- `elements_check`

## Field Rules

### verdict

必须是四类之一：

- `适合长期使用`
- `可以继续使用，但要注意使用方式`
- `不建议继续长期主用`
- `不建议长期使用，请尽快调整`

### content

用户可见的一段话。

要求：
- 开头或结尾必须明确体现 verdict 的意思。
- 只写一段，不拆条，不列表。
- 说明为什么能长期用或为什么不建议长期用。
- 必须说明长期使用会落到哪些用户能对照的现实场景，例如主号承接、合作收口、关系维护、资金沉淀、精力消耗或出行安排。
- 必须结合最关键的稳定性扣分项。
- 如果用户明确提供关注项，要说明该关注项是否影响长期使用结论。
- 不要写成内部评分档位或内部评分语言。
- 不要连续堆叠专业术语；如果使用 `空亡 / 门迫 / 入墓 / 击刑 / 特殊组合 / 四害` 等词，必须同时写出它在现实里怎么表现。
- 不要提课程、知识库、文档、老师或来源。

### elements_check

内部结构化判断对象，必须包含：

- `整体承接`
- `宫门关系`
- `四害`
- `特殊组合`
- `神星门`
- `天干/地干`
- `风险敏感方向`
- `最终使用结论`

每项用一句话说明，不要空值。
每项都要写成“专业层 + 现实含义”，不要只写标签。

## Minimum Quality Bar

用户读完 content 后，至少应该知道：
- 这个号能不能长期用
- 如果能用，是不是适合当主号
- 如果不能主用，问题主要在哪里
- 如果要继续用，应该避开什么场景或怎么调整
- 如果用户明确提供关注项，这些关注项会不会改变长期使用结论；如果没有明确关注项，只说明盘面风险敏感方向
