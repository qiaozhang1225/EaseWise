# Shared User-Readable Expression

## Goal

手机号评测的用户可见文案，要让用户读完能马上拿自己的经历做对照。

最终输出不是给专业人士复盘盘面，而是回答三个问题：

- 我平时是不是这种状态
- 这个号码的问题会落在哪些具体场景
- 我接下来应该怎么使用或规避

## Translation Rule

任何专业判断都必须翻译成现实表现。

可以少量保留术语，但术语不能当结论。只要出现 `宫`、`门`、`神`、`星`、`天干/地干`、`空亡`、`门迫`、`入墓`、`击刑`、`特殊组合`、`四害` 等词，必须紧跟它对应的用户能理解的含义，例如：

- 不只写“门迫”，要写成“推进时容易自己和环境互相顶住，有力使不出”。
- 不只写“空亡”，要写成“看起来有机会，但到执行时容易落不实”。
- 不只写“入墓”，要写成“前面能启动，后面收口慢、钱或结果压住不动”。
- 不只写“击刑”，要写成“容易在关键处出错、受压、别扭或临时被打断”。

## Public Paragraph Rules

`title`、`risk`、`usage_guidance`、`content` 这些用户主段落必须：

- 先说结论，再解释原因。
- 先写用户能核实的表现，再少量带出判断来源。
- 用生活化场景表达，例如项目推进、合作分工、回款、关系沟通、睡眠恢复、考试发挥、出行安排。
- 每段最多自然带出 1-2 个专业词；如果专业词会让句子变硬，就不要写术语，只写现实表现。
- 不要用术语开头或结尾。
- 不要把 `elements_check` 里的七层判断逐条抄进主段落。
- 不要把“分数、结构、格局、盘面、四害、特殊组合”写成用户必须理解的前置知识。

## Self-Check Standard

生成前自检：

- 用户能不能用“像不像我”快速判断准确率。
- 有没有至少一个具体场景，而不是只有抽象词。
- 风险是不是说清楚了“会怎么发生”，而不是只说“有风险”。
- 建议是不是可执行，例如适合主用、辅助用、谨慎合作、减少重仓、先观察、提前核对。
- 是否避免了专业词连续堆叠。

## Elements Check Rule

`elements_check` 可以保留结构化分层，因为它用于系统追踪判断。

但每项也要写成“专业层 + 现实含义”的短句，不要只写标签解释。例如：

- 好：`门这一层显示行动会受压，现实里容易出现想推进但被流程、合作或自身节奏卡住。`
- 差：`门迫，行动受压。`
