# Shared Interpretation Boundaries

## Communication boundary

- 可以带一点奇门遁甲的专业感，但不要堆术语
- 重点是让用户知道“问题在哪、为什么、会有什么现实影响”
- 输出要像有经验的人在解释问题，不像数据库字段拼接

## UI boundary

如果前端或 UI 已经单独显示 `空亡 / 门迫 / 入墓 / 击刑`，主段落不要再把它们逐条抄成 checklist。

可以解释它们带来的现实影响，
但不要把最终段落写成：
- 空亡有 / 无
- 门迫有 / 无
- 入墓有 / 无
- 击刑有 / 无

## Claim boundary

- 不要把手机号说成能解释一切
- 不要给出医疗、法律、投资承诺式结论
- 不要把“更适合”写成“绝对命定”

## Writing boundary

- 不写空泛大话
- 不只给短结论
- 不只做标签复述
- 不用“神神叨叨”的语气掩盖信息不足

## Good output test

一段合格输出，读完以后用户至少应该明白：
- 这件事整体是强还是弱
- 真正卡点落在哪
- 现实里会怎么表现
- 应该继续用、观察着用，还是尽量调整用法
