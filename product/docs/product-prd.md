# 易如反掌 产品 PRD

- Status: active draft
- Phase: 新仓库重建期 / H5 验证优先 / 微信生态优先
- Baseline date: 2026-05-23
- External product name: `易如反掌`
- English product name: `EaseWise`
- Current repository name: `EaseWise`
- Source of truth: `product/docs/product-prd.md`、`product/docs/product-cards.md`

## 1. 项目背景

### 1.1 命名与项目定位

当前项目正式对外产品名统一为 `易如反掌`，英文名为 `EaseWise`。

当前阶段，这个项目的定位不是单一工具页，而是一个以 `AI + 传统文化 / 玄学能力` 为基础、面向移动端的轻量产品体系。项目会先用“手机号评测”建立第一个清晰、可理解、可转化的用户入口，再逐步扩展其他能力。

`易如反掌` 的长期目标是以 `AI + 传统文化 / 玄学能力` 为基础，为用户提供：

- 决策辅助
- 情绪价值
- 可持续复访的个人工具体验

`奇门手机号评测` 是 `易如反掌` 面向用户的第一个实用型能力，也是当前阶段最重要的增长与商业化入口。

### 1.2 当前阶段为什么仍以 H5 优先

当前阶段虽然先做 H5，但产品不是为单一 H5 页面而生，而是按阶段推进：

- 前期：以 H5 验证完整业务闭环
- 中期：以微信小程序为主承接传播与使用
- 后期：以 App 承接长期留存与更完整能力

当前在新仓库里，具体前端技术栈与工程结构还没有最终锁定；但产品交付方向仍然保持不变：

- H5 阶段可以更快验证产品闭环
- 前端结构应为后续小程序迁移预留空间
- 后续如需推进 App，应尽量复用已有移动端交互范式
- 当前明确不以桌面 Web 体验为优先，而以手机端体验为优先

换句话说，`H5 优先` 是当前产品路径，`具体技术栈` 则留给新仓库的架构设计阶段确定。

## 2. 产品定位

`易如反掌` 当前不是一个“大而全”的综合玄学平台，而是一个以可快速体验、可持续转化为目标的移动端产品。

当前阶段的产品定义可以压缩为一句话：

> 用低门槛的 AI 评测能力吸引用户进入，再通过积分、私域和后续能力矩阵把一次性兴趣转化为持续使用。

当前阶段最重要的产品心智是：

- 这是一个很准、很有用的决策辅助工具
- 手机号评测是第一个用户最容易理解、最容易付费的入口
- 后续会逐渐扩展为多个 AI + 传统文化能力共同构成的产品矩阵

## 3. 当前阶段目标与衡量指标

### 3.1 当前阶段目标

H5 阶段的目标不是堆很多能力，而是验证一个完整、可复用、可转化的产品闭环：

1. 用户愿意测
2. 用户愿意付费
3. 用户愿意加私域

### 3.2 当前阶段核心指标

当前优先级最高的指标为：

- AI 付费转化率
- 私域添加率
- 复访率

当前阶段暂不把“智能体使用率”设为核心 KPI。智能体在本阶段保留基础对话能力即可，不承担主要转化目标。

## 4. 目标用户与核心需求

### 4.1 目标用户

当前阶段的核心目标用户是：

- 对玄学、传统文化、命理内容感兴趣的人
- 愿意尝试低门槛 AI 服务的人
- 对个人状态、决策、关系、财运、事业等问题存在关注的人
- 可能进一步接受 1v1 服务的人

### 4.2 用户在手机号评测里的核心问题

用户在“奇门手机号评测”里最想知道的通常是：

- 这个手机号到底好不好
- 这个手机号适不适合自己长期使用
- 它会不会对自己当前在意的事情产生正面或负面影响
- 是否值得找老师进一步定制更适合自己的手机号

## 5. 产品结构

### 5.1 首页

首页是能力入口页，但在当前阶段需要让“手机号评测”更突出。

首页结构如下：

- 顶部：黄历展示，用于建立传统文化氛围，不是核心商业能力
- 中部：
  - 1 个主推入口：手机号解析
  - 6 个次级能力入口：
    - 头像解析
    - 姓名解析
    - 八字解析
    - 奇门问事
    - 黄历查询
    - 五行属性
- 下部：移动端底部导航

首页策略：

- 形态上是“能力矩阵”
- 运营上突出 1 个能力，当前优先突出“手机号解析”
- 其余入口现阶段可以先保留占位或轻量提示，后续再逐步实装

补充说明：

- 黄历不是核心产品能力，只承担氛围、日常感知和相对安全的内容入口角色
- 为了后续小程序合规稳定，可以保留 1~2 个偏安全的工具能力，例如：黄历查询、五行属性

### 5.2 手机号评测页

手机号评测页是当前阶段最重要的产品页。

#### 输入

仅保留两个输入项：

- 手机号
- 性别

#### 交互

- 点击“评测”后，不跳转结果页
- 评测页与结果页是同一页
- 用户在原地等待并查看结果展开

#### 结果结构

结果需要按以下结构组织：

1. 评分
2. 盘面展示
3. 盘面分析 / 总评
4. 长期使用建议 / 稳定性判断
5. 九个独立维度结果

补充说明：

- 产品判断层面仍然保留完整的“十方面”认知
- 其中“稳定性”不再作为一个单独解锁维度卡片，而是并入基础结果区，与总评、长期使用建议一起首屏展示
- 其余九个维度继续以折叠、标签或分段切换方式承接，保持当前移动端阅读效率

#### 页面要求

- 评分 + 盘面 + 总评应控制在手机端约 1 到 1.5 屏内
- 用户轻微下滑后就应看到“长期使用建议 / 稳定性判断”以及后续九个维度内容
- 九个独立维度建议采用折叠、标签或分段切换等方式，避免页面过长
- 若后续支持“导出图片”，则导出态可以展开为完整长图

#### 等待体验

手机号评测生成需要明确降低用户等待焦虑：

- 前端要展示阶段性进度感
- 用户应知道系统正在工作，而不是页面卡住
- 即使后端仍是同步生成，也要提供足够明确的等待反馈

### 5.3 智能体页

智能体页在当前阶段保留为基础聊天页。

当前阶段要求：

- 提供最基本的大模型连续对话能力
- 不接入手机号评测、头像、梅花等工具能力
- 不作为当前阶段主转化入口

中长期方向：

- 智能体将成为“情绪价值 + 决策辅助”的主要入口
- 首页上的各类功能未来可逐渐转为智能体可调用的工具
- 但这属于后续大版本目标，不属于当前 PRD 的已批准交付范围

### 5.4 我的页面

“我的”页面当前阶段优先级如下：

1. 登录 / 注册
2. 当前剩余积分与充值入口
3. 推广大使入口
4. 历史记录
5. 反馈问题
6. 登录 / 退出登录按钮

页面内容应包括：

- 顶部登录信息
- 登录信息下方展示当前剩余积分
- 一个充值按钮
- 一列入口按钮：充值记录、评测记录、反馈问题
- 最下方立即登录 / 退出登录

## 6. 商业模式与积分体系

### 6.1 当前阶段商业重点

当前阶段最重要的转化点是购买 AI 服务。

原因：

- 金额最低
- 决策门槛最低
- 是用户进入产品商业闭环的第一步
- 能验证这个产品是否具备基础收入能力
- 能为后续私域和 1v1 服务转化沉淀用户资产

1v1 服务存在明确商业价值，但当前阶段不作为前台主推，而更多放在私域内承接。

### 6.2 积分

当前阶段不存在“无限免费 AI 服务”。

所谓免费，本质上是通过各种方式给用户发放 `积分`，再由用户消耗积分体验服务。积分来源包括：

- 登录 / 注册
- 邀请他人
- 完成任务
- 完善资料
- 运营发放
- 充值获得

当前方向上：

- 基础评测暂按 `100` 积分理解
- 每解锁一个额外维度暂按 `50` 积分理解

但上述数值只是产品方向，不是最终定价，必须支持后台配置。

新用户默认获得的积分，应大致足够：

- 完成一次基础评测
- 再额外查看 1~2 个维度，或体验一次其他轻量能力

补充说明：

- 基础评测结果中默认包含评分、盘面、总评，以及“长期使用建议 / 稳定性判断”
- 付费解锁主要针对剩余九个独立维度，不要求把“稳定性”再做成单独解锁项

### 6.3 充值策略

在 H5 与小程序阶段：

- 产品内不直接提供常规充值闭环
- 用户点击“充值”后，应被强引导去联系私域客服
- 客服再发送独立 H5 充值链接

这样做的目的包括：

- 降低小程序审核与支付相关风险
- 提高私域沉淀与后续转化率
- 为多个小程序矩阵提供更灵活的运营空间

App 阶段是否直接内建充值，后续再决定。

## 7. 推广大使与推广收益

### 7.1 基本原则

前台产品表达中，`积分` 与 `推广收益` 是两套不同的资产概念，必须分开展示和管理。

统一定义如下：

- `积分`：用于普通功能消费的唯一积分资产
- `推广收益`：由推广返佣产生的资金型收益资产，只用于提现和资金结算，不再作为积分使用

后台管理与账务层面，用户资产统一拆为：

- `积分`
- `可提现余额`
- `冻结中返佣余额`
- `返佣记录`
- `提现记录`

前台默认只展示积分。只有用户进入“推广大使”相关页面后，才展示推广收益中的可提现余额、冻结中返佣余额及对应现金额度。

### 7.2 推广大使机制

系统内推广身份仅分为两级：

- 普通推广大使
- 高级推广大使

两者区别仅在于直邀用户订单完成后的返佣比例不同，不存在团队层级收益。

当前准入规则定义为：

- 用户累计充值金额达到 `398 元` 后，可申请成为普通推广大使
- 用户累计充值金额达到 `3980 元` 后，可申请成为高级推广大使

以上门槛数值、返佣比例均为后台可配置项。

补充说明：

- 门槛配置为全局统一配置
- 配置修改后对全部用户生效
- 订单层面的返佣计算规则以订单创建时或订单结算时的规则快照为准，只影响后续新订单
- 管理员可在后台直接调整用户推广身份

### 7.3 申请、审核与身份变更

推广身份不自动生效，用户达到门槛后仍需主动申请，并由后台审核。

审核规则如下：

- 审核动作为 `通过` 或 `拒绝`
- 审核人必须能看到申请人的姓名、手机号等基础信息
- 拒绝时必须填写拒绝原因
- 已拒绝后允许再次申请
- 普通推广大使可继续申请升级为高级推广大使
- 管理员可直接手动调整用户推广身份

### 7.4 绑定关系与归属规则

用户与推广大使之间通过邀请码或推广链接建立绑定关系。

绑定规则如下：

- 一个用户只允许有一个上级推广大使
- 用户一旦绑定上级，默认永久归属
- 用户本人不能自行修改上级归属
- 管理员可在后台手动绑定或更换用户上级
- 所有手动调整归属的操作都必须保留操作记录

### 7.5 推广返佣规则

所有充值订单都参与返佣，当前不包含其他商品类型。

返佣产生规则如下：

- 返佣按用户订单的实付金额计算
- 订单支付后可先形成待结算返佣记录，并计入冻结中返佣余额
- 订单在 `已完成` 后，冻结中的返佣结算为可提现余额
- 若订单后续发生退款，已产生或已结算的返佣必须被撤回
- 撤回返佣时，优先冲减冻结中返佣余额；已结算部分从可提现余额中冲减
- 若撤回后推广资金账户出现负数，也允许保留负数余额并在后续收益中抵扣

订单 `已完成` 的定义如下：

- 用户支付后，在配置观察期内产生与该充值相关的积分消费记录，则该订单提前视为 `已完成`
- 若在观察期内未产生积分消费记录，则支付满观察期后自动视为 `已完成`

默认观察期可按 `7 天` 理解，但该天数必须支持后台配置。

### 7.6 推广收益的使用规则

推广收益只作为资金型资产使用，不再作为积分资产参与功能消费。

使用规则如下：

- 可提现余额可用于发起提现申请
- 冻结中返佣余额不可提现，需等待订单完成和返佣结算
- 推广收益不支持转换为积分
- 当用户积分不足时，不能使用推广收益抵扣功能消费
- 所有返佣结算、提现、退款撤回都必须保留资金流水记录

### 7.7 提现规则

只有推广大使或高级推广大使才具备提现权限。

提现规则如下：

- 用户可在前台主动发起提现申请
- 前台必须展示：可提现余额、冻结中返佣余额、最低提现门槛、提现金额输入框、`全部提现` 按钮
- 最低提现门槛当前按 `30 元` 执行，并支持后台配置
- 用户提交提现申请后，由后台审核
- 审核通过后，系统自动发起分账打款
- 若自动打款失败，系统进入内部失败处理流程，由管理员决定重试分账或手动完成打款
- `打款失败` 属于后台内部状态，不对普通用户展示

补充说明：

- 提现时不要求用户再次勾选协议
- 用户成为推广大使时，应在前置流程中完成简易协议确认
- 最低提现门槛和订单完成判定天数应支持后台配置

## 8. 登录、身份与权限

### 8.1 用户身份体系目标

后续用户体系以手机号作为正式主账号，以 UnionID 作为微信生态下的统一识别字段。

产品目标是：

- 用户未来进入 App 时，可以通过手机号登录，并继承其在 H5 或小程序阶段产生的资产
- 用户从微信 H5、公众号、小程序或多个小程序进入时，只要 UnionID 相同，就应命中同一个系统用户
- 同一用户的积分、可提现余额、冻结中返佣余额、订单、使用记录、返佣与提现记录应保持连续
- Session 只作为临时访问态或登录态承载，不作为长期用户身份

补充说明：

- UnionID 是微信体系内的主要业务识别字段
- OpenID 只作为单个微信应用内的技术标识保留，可用于排查，不作为产品主身份

### 8.2 游客与登录用户的能力边界

未登录用户可以：

- 查看当前页面和当前会话中的临时状态
- 进行不依赖正式账户资产的基础浏览
- 在登录引导前了解产品能力

未登录用户不可以：

- 领取免费初始积分
- 使用智能体
- 充值
- 使用推广大使能力
- 通过任务获取更多积分

登录 / 注册后，用户才能进入完整账户体系。

### 8.3 免费初始积分领取规则

免费初始积分不再按 Session 发放。

规则如下：

- 用户通过 Session 访问时，系统仅展示当前页面状态
- 前台可提示用户 `登录以获取 XX 积分`
- 用户完成手机号验证或微信 UnionID 登录后，才可领取一次免费初始积分
- 同一手机号或同一 UnionID 只能领取一次免费初始积分
- 当前阶段暂不实现阿里云短信验证、微信 UnionID 登录和领取流程，先作为产品规则预留

## 9. 私域引导策略

当前阶段，私域添加是核心目标之一。

引导优先级如下：

1. 任何未被判定为“已加客服”的用户，都应被提示加客服
2. 积分不足时，提示联系客服充值
3. 用户主动点击充值时，进行强引导或直接跳转

当前判定规则可暂按：

- 若该微信身份没有使用过“免费领取积分”能力，则视为尚未进入私域

V1 原则：

- 关键节点要引导
- 但不要通过过多弹窗破坏体验
- 结果看完后的额外弹窗引导，暂不作为第一版强制项

## 10. 合规与安全开关

后台必须支持按渠道进行合规与安全控制。

这里的“渠道”包括但不限于：

- `H5`
- 每一个独立小程序
- `App`

每个渠道都应具备自己的总开关。总开关关闭后，当前渠道内所有玄学相关能力都应关闭，包括但不限于：

- 手机号评测
- 四柱八字
- 其他玄学分析能力
- 智能体中的玄学相关技能

总开关关闭后，以下能力不受影响：

- 首页顶部黄历卡片
- 黄历查询功能
- 五行属性查询功能
- 智能体基础聊天功能
- “我的”页面相关功能

除渠道总开关外，每个玄学功能还应具备独立开关。

补充规则：

- 智能体开关不是关闭整个智能体，而是关闭其玄学技能调用能力
- 当某个功能被关闭后，前台应隐藏对应入口
- 当某个功能被关闭后，与该功能相关的历史使用记录也应一并隐藏
- 渠道总开关关闭时，同样适用入口隐藏与历史记录隐藏规则

## 11. 后台与运营配置要求

后台系统至少覆盖以下两类配置能力：

### 11.1 功能管理类配置

功能管理类配置主要承接具体业务功能的开关、显隐与积分规则。

至少包括：

- 手机号评测
- 智能体玄学技能
- 其他后续玄学能力

其中，手机号评测需要支持：

- 基础评测积分消耗配置
- 九个独立维度的解锁积分配置
- 免费维度配置
- 维度展示顺序配置
- 功能开关与渠道显隐控制

智能体需要支持：

- 玄学技能开关
- 渠道显隐控制
- 与总开关联动的能力限制

### 11.2 系统级配置

系统级配置主要承接全局基础规则，不承接具体业务功能的细节规则。

至少包括：

- 充值套餐配置
- 过渡期游客初始积分配置
- 注册用户初始积分配置
- 客服联系方式配置
- 少量平台级总开关配置

补充说明：

- 过渡期游客初始积分配置仅用于手机号短信验证或微信 UnionID 登录上线前的兼容，后续应停用或调整为 0
- 推广大使门槛、返佣比例、提现门槛、订单完成判定天数，不放在通用系统配置中，而放在后台 `推广合作 > 规则配置`
- 手机号评测、智能体等具体功能规则，不放在通用系统配置中，而放在后台 `功能管理`
- 后台必须支持通过表单方式维护这些配置，不要求运营人员直接操作底层配置结构

## 12. 阶段推进路径

### 12.1 H5 阶段

目标：

- 快速验证完整闭环
- 降低发布与审核阻力
- 降低“产品被投诉后临时下线”的风险
- 在移动端先把产品流程做顺

### 12.2 小程序阶段

进入条件：

- 产品流程稳定
- 后台系统与运营配置稳定

小程序阶段目标：

- 提升使用体验
- 提升传播效率
- 通过多个小程序矩阵共用同一套前端代码与后端服务，降低单点风险
- 更好地承接分享、传播与私域转化

### 12.3 App 阶段

App 是更远期的形态，主要面向：

- 长期留存用户
- 需要更完整、更高频能力的用户

是否在第一版 App 中沿用小程序逻辑，还是直接设计新的产品逻辑与前端架构，当前不做锁定，需在小程序阶段跑通后再决策。

## 13. 能力扩展优先级

在手机号评测之后，当前优先顺序为：

1. 头像解析
2. 姓名解析
3. 四柱八字
4. 奇门问事
