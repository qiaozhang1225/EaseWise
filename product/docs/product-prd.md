# 易如反掌 产品 PRD

- Status: active draft
- Phase: 新仓库重建期 / H5 验证优先 / 微信生态优先
- Baseline date: 2026-05-23
- External product name: `易如反掌`
- English product name: `EaseWise`
- Current repository name: `EaseWise`
- Source of truth: `product/docs/product-prd.md`、`product/docs/product-cards.md`

## 1. 项目背景

### 1.1 命名与项目定位

当前项目正式对外产品名统一为 `易如反掌`，英文名为 `EaseWise`。

当前阶段，这个项目的定位不是单一工具页，而是一个以 `AI + 传统文化 / 玄学能力` 为基础、面向移动端的轻量产品体系。项目会先用“手机号评测”建立第一个清晰、可理解、可转化的用户入口，再逐步扩展其他能力。

`易如反掌` 的长期目标是以 `AI + 传统文化 / 玄学能力` 为基础，为用户提供：

- 决策辅助
- 情绪价值
- 可持续复访的个人工具体验

`奇门手机号评测` 是 `易如反掌` 面向用户的第一个实用型能力，也是当前阶段最重要的增长与商业化入口。

### 1.2 当前阶段为什么仍以 H5 优先

当前阶段虽然先做 H5，但产品不是为单一 H5 页面而生，而是按阶段推进：

- 前期：以 H5 验证完整业务闭环
- 中期：以微信小程序为主承接传播与使用
- 后期：以 App 承接长期留存与更完整能力

当前在新仓库里，具体前端技术栈与工程结构还没有最终锁定；但产品交付方向仍然保持不变：

- H5 阶段可以更快验证产品闭环
- 前端结构应为后续小程序迁移预留空间
- 后续如需推进 App，应尽量复用已有移动端交互范式
- 当前明确不以桌面 Web 体验为优先，而以手机端体验为优先

换句话说，`H5 优先` 是当前产品路径，`具体技术栈` 则留给新仓库的架构设计阶段确定。

## 2. 产品定位

`易如反掌` 当前不是一个“大而全”的综合玄学平台，而是一个以可快速体验、可持续转化为目标的移动端产品。

当前阶段的产品定义可以压缩为一句话：

> 用低门槛的 AI 评测能力吸引用户进入，再通过积分、私域和后续能力矩阵把一次性兴趣转化为持续使用。

当前阶段最重要的产品心智是：

- 这是一个很准、很有用的决策辅助工具
- 手机号评测是第一个用户最容易理解、最容易付费的入口
- 后续会逐渐扩展为多个 AI + 传统文化能力共同构成的产品矩阵

## 3. 当前阶段目标与衡量指标

### 3.1 当前阶段目标

H5 阶段的目标不是堆很多能力，而是验证一个完整、可复用、可转化的产品闭环：

1. 用户愿意测
2. 用户愿意付费
3. 用户愿意加私域

### 3.2 当前阶段核心指标

当前优先级最高的指标为：

- AI 付费转化率
- 私域添加率
- 复访率

当前阶段暂不把“智能体使用率”设为核心 KPI。智能体在本阶段保留基础对话能力即可，不承担主要转化目标。

## 4. 目标用户与核心需求

### 4.1 目标用户

当前阶段的核心目标用户是：

- 对玄学、传统文化、命理内容感兴趣的人
- 愿意尝试低门槛 AI 服务的人
- 对个人状态、决策、关系、财运、事业等问题存在关注的人
- 可能进一步接受 1v1 服务的人

### 4.2 用户在手机号评测里的核心问题

用户在“奇门手机号评测”里最想知道的通常是：

- 这个手机号到底好不好
- 这个手机号适不适合自己长期使用
- 它会不会对自己当前在意的事情产生正面或负面影响
- 是否值得找老师进一步定制更适合自己的手机号

## 5. 产品结构

### 5.1 首页

首页是能力入口页，但在当前阶段需要让“手机号评测”更突出。

首页结构如下：

- 顶部：黄历展示，用于建立传统文化氛围，不是核心商业能力
- 中部：
  - 1 个主推入口：手机号解析
  - 6 个次级能力入口：
    - 头像解析
    - 姓名解析
    - 八字解析
    - 奇门问事
    - 黄历查询
    - 五行属性
- 下部：移动端底部导航

首页策略：

- 形态上是“能力矩阵”
- 运营上突出 1 个能力，当前优先突出“手机号解析”
- 其余入口现阶段可以先保留占位或轻量提示，后续再逐步实装

补充说明：

- 黄历不是核心产品能力，只承担氛围、日常感知和相对安全的内容入口角色
- 为了后续小程序合规稳定，可以保留 1~2 个偏安全的工具能力，例如：黄历查询、五行属性

### 5.2 手机号评测页

手机号评测页是当前阶段最重要的产品页。

#### 输入

仅保留两个输入项：

- 手机号
- 性别

#### 交互

- 点击“评测”后，不跳转结果页
- 评测页与结果页是同一页
- 用户在原地等待并查看结果展开

#### 结果结构

结果需要按以下结构组织：

1. 评分
2. 盘面展示
3. 盘面分析 / 总评
4. 长期使用建议 / 稳定性判断
5. 九个独立维度结果

补充说明：

- 产品判断层面仍然保留完整的“十方面”认知
- 其中“稳定性”不再作为一个单独解锁维度卡片，而是并入基础结果区，与总评、长期使用建议一起首屏展示
- 其余九个维度继续以折叠、标签或分段切换方式承接，保持当前移动端阅读效率

#### 页面要求

- 评分 + 盘面 + 总评应控制在手机端约 1 到 1.5 屏内
- 用户轻微下滑后就应看到“长期使用建议 / 稳定性判断”以及后续九个维度内容
- 九个独立维度建议采用折叠、标签或分段切换等方式，避免页面过长
- 若后续支持“导出图片”，则导出态可以展开为完整长图

#### 等待体验

手机号评测生成需要明确降低用户等待焦虑：

- 前端要展示阶段性进度感
- 用户应知道系统正在工作，而不是页面卡住
- 即使后端仍是同步生成，也要提供足够明确的等待反馈

### 5.3 智能体页

智能体页在当前阶段保留为基础聊天页。

当前阶段要求：

- 提供最基本的大模型连续对话能力
- 不接入手机号评测、头像、梅花等工具能力
- 不作为当前阶段主转化入口

中长期方向：

- 智能体将成为“情绪价值 + 决策辅助”的主要入口
- 首页上的各类功能未来可逐渐转为智能体可调用的工具
- 但这属于后续大版本目标，不属于当前 PRD 的已批准交付范围

### 5.4 我的页面

“我的”页面当前阶段优先级如下：

1. 登录 / 注册
2. 当前剩余积分与充值入口
3. 推广大使入口
4. 历史记录
5. 反馈问题
6. 登录 / 退出登录按钮

页面内容应包括：

- 顶部登录信息
- 登录信息下方展示当前剩余积分
- 一个充值按钮
- 一列入口按钮：充值记录、评测记录、反馈问题
- 最下方立即登录 / 退出登录

## 6. 商业模式与积分体系

### 6.1 当前阶段商业重点

当前阶段最重要的转化点是购买 AI 服务。

原因：

- 金额最低
- 决策门槛最低
- 是用户进入产品商业闭环的第一步
- 能验证这个产品是否具备基础收入能力
- 能为后续私域和 1v1 服务转化沉淀用户资产

1v1 服务存在明确商业价值，但当前阶段不作为前台主推，而更多放在私域内承接。

### 6.2 服务积分

当前阶段不存在“无限免费 AI 服务”。

所谓免费，本质上是通过各种方式给用户发放 `服务积分`，再由用户消耗积分体验服务。服务积分来源包括：

- 登录 / 注册
- 邀请他人
- 完成任务
- 完善资料
- 运营发放
- 充值获得

当前方向上：

- 基础评测暂按 `100` 积分理解
- 每解锁一个额外维度暂按 `50` 积分理解

但上述数值只是产品方向，不是最终定价，必须支持后台配置。

新用户默认获得的积分，应大致足够：

- 完成一次基础评测
- 再额外查看 1~2 个维度，或体验一次其他轻量能力

补充说明：

- 基础评测结果中默认包含评分、盘面、总评，以及“长期使用建议 / 稳定性判断”
- 付费解锁主要针对剩余九个独立维度，不要求把“稳定性”再做成单独解锁项

### 6.3 充值策略

在 H5 与小程序阶段：

- 产品内不直接提供常规充值闭环
- 用户点击“充值”后，应被强引导去联系私域客服
- 客服再发送独立 H5 充值链接

这样做的目的包括：

- 降低小程序审核与支付相关风险
- 提高私域沉淀与后续转化率
- 为多个小程序矩阵提供更灵活的运营空间

App 阶段是否直接内建充值，后续再决定。

## 7. 推广大使与推广收益

### 7.1 基本原则

`推广收益` 与 `服务积分` 不是同一个概念，产品表达上应分开。

建议当前文档先使用：

- `服务积分`
- `推广收益`

最终对用户展示时，如需改名，可在后续版本统一调整。

### 7.2 推广大使机制

正常用户默认不具备可提现推广收益能力。

用户在累计消费达到一定门槛后，可开启推广身份：

- 推广大使：暂定消费门槛 `398 元`
- 高级推广大使：暂定消费门槛 `3980 元`

这两个门槛及其返佣比例都必须支持后台配置。

当前仅确认产品方向，不锁定最终数值。

### 7.3 交互规则

- 用户未成为推广大使前：入口点击后优先引导加客服
- “我的”页面不直接写明“充值 398 成为推广大使”
- 可通过轻提示方式表达“成为推广大使赚佣金”
- 成为推广大使后：入口进入推广页，展示推广记录、收益与提现能力
- 普通推广大使页面中可展示升级到高级推广大使的进度条

### 7.4 推广收益的使用方向

推广收益：

- 可以提现
- 也可以转成服务积分

产品方向上，应优先鼓励用户把推广收益转成服务积分继续消费，而不是直接提现。

是否在服务积分不足时自动优先消耗推广收益，当前仍未最终决定。

## 8. 登录、身份与权限

### 8.1 微信生态下的游客连续性

当前产品运行在微信生态中，即使是 H5，也应尽量利用微信体系内的身份连续性能力。

产品目标是：

- 用户即使没有正式登录 / 注册，也可能基于微信侧身份拿到历史数据连续性
- 用户再次打开 H5 或未来切到小程序时，仍能识别其历史状态

### 8.2 游客与登录用户的能力边界

未登录用户可以：

- 使用默认赠送的服务积分
- 进行基础体验
- 在一定范围内保留历史记录连续性

未登录用户不可以：

- 使用智能体
- 充值
- 使用推广大使能力
- 通过任务获取更多积分

登录 / 注册后，用户才能进入完整账户体系。

## 9. 私域引导策略

当前阶段，私域添加是核心目标之一。

引导优先级如下：

1. 任何未被判定为“已加客服”的用户，都应被提示加客服
2. 服务积分不足时，提示联系客服充值
3. 用户主动点击充值时，进行强引导或直接跳转

当前判定规则可暂按：

- 若该微信身份没有使用过“免费领取积分”能力，则视为尚未进入私域

V1 原则：

- 关键节点要引导
- 但不要通过过多弹窗破坏体验
- 结果看完后的额外弹窗引导，暂不作为第一版强制项

## 10. 合规与安全开关

这是一个非常重要的系统级能力。

一个后台未来可能同时服务多个小程序或多个前端壳，因此必须支持：

- 全局级一键开关
- 按小程序 / 渠道的一键开关
- 按页面 / 模块的一键开关

当某个小程序因玄学内容被举报或要求整改时，应能快速：

- 隐藏玄学相关内容
- 保留相对安全的工具能力
- 重新提审
- 审核通过后再恢复相应能力

当前可保留的安全能力方向：

- 黄历查询
- 五行属性查询

## 11. 后台与运营配置要求

后台系统至少需要覆盖两层配置能力。

### 11.1 功能模块配置

每个功能都应有独立配置页或独立标签页，例如：

- 手机号评测
- 梅花易数
- 四柱八字
- 其他后续能力

每个功能模块都应允许配置：

- 积分消耗规则
- 免费 / 付费解锁规则
- 展示顺序
- 运营策略相关开关

其中“手机号评测”模块应视为特殊模块，支持更细粒度的配置，例如：

- 九个独立维度顺序调整
- 某个方面提前展示
- 某个方面免费解锁
- “长期使用建议 / 稳定性判断”是否强化展示
- 某些内容用于引导 1v1 转化

### 11.2 系统级配置

系统模块应包含但不限于：

- 充值套餐配置
- 积分规则配置
- 新用户赠送规则配置
- 任务得分规则配置
- 推广大使门槛与返佣比例配置
- 客服二维码或跳转链接配置
- 手动加积分 / 调整用户积分能力
- 用户管理
- 订单与充值记录管理
- 工具模式 / 产品模式等系统级开关

## 12. 阶段推进路径

### 12.1 H5 阶段

目标：

- 快速验证完整闭环
- 降低发布与审核阻力
- 降低“产品被投诉后临时下线”的风险
- 在移动端先把产品流程做顺

### 12.2 小程序阶段

进入条件：

- 产品流程稳定
- 后台系统与运营配置稳定

小程序阶段目标：

- 提升使用体验
- 提升传播效率
- 通过多个小程序矩阵共用同一套前端代码与后端服务，降低单点风险
- 更好地承接分享、传播与私域转化

### 12.3 App 阶段

App 是更远期的形态，主要面向：

- 长期留存用户
- 需要更完整、更高频能力的用户

是否在第一版 App 中沿用小程序逻辑，还是直接设计新的产品逻辑与前端架构，当前不做锁定，需在小程序阶段跑通后再决策。

## 13. 能力扩展优先级

在手机号评测之后，当前优先顺序为：

1. 头像解析
2. 姓名解析
3. 四柱八字
4. 奇门问事

## 14. 待确认事项

以下内容已形成方向，但仍需后续继续定稿：

- 服务积分与推广收益的最终对外命名
- 基础评测与额外解锁的最终积分定价
- 推广大使返佣比例与门槛数值
- 推广收益转积分与提现换算公式
- 积分不足时是否自动优先消耗推广收益
- 手机号评测结果导出长图的具体版式

## 15. 文档治理规则

- 在 `EaseWise` 新仓库中，产品定义优先沉淀在 `product/docs/`，再进入实现阶段
- 产品背景补充、术语澄清、阶段说明，可直接更新到 `PRD`
- 任何影响产品范围、核心交互、积分 / 定价、平台策略、发布形态的重要变化，都应先落到 `product card`，再进入执行计划
- `PRD` 记录产品真相与目标方向
- 新仓库的 `spec` 需在实现结构明确后再建立，用于记录实际交付基线
