- Status: active draft

## 1. 背景与问题

当前 `易如反掌` 的用户前台已经可以通过本地 API 进行基础测试，但测试与运营工作仍然非常低效。

最直接的问题包括：

- 初始积分、评测扣分、解锁扣分等规则只能通过环境变量或内部配置接口调整
- 测试过程中如果想把“游客初始积分”从 `100` 临时改成 `500` 或 `1000`，目前没有可视化后台入口
- 客服联系方式、黄历安全模块开关、渠道级配置也缺少统一管理入口
- 用户积分调整、充值审核、评测记录排查、使用记录排查没有后台页面承接
- 当前虽然已有一部分 `internal API`，但主要靠请求头 token 直接调接口，不适合日常测试和运营使用

因此，后台管理系统的第一目标不是“做一个完整企业后台”，而是：

> 先做一套能显著提升本地测试效率、运营配置效率和问题排查效率的内部工作台。

## 2. 产品定位

`易如反掌后台` 是服务于内部测试、运营、客服和后续财务审核的桌面端内部系统。

它不是面向普通用户的产品能力，而是产品闭环的运营控制台。

当前阶段的定位可以压缩为一句话：

> 用最少的后台页面，接住当前已经存在的内部接口能力，让测试、运营和客服不用再改环境变量、查数据库、手敲接口。

## 3. 当前阶段目标

### 3.1 V1 目标

V1 先解决以下四类问题：

1. 能快速改配置  
2. 能快速查用户  
3. 能快速调积分  
4. 能快速处理充值审核与问题排查  

### 3.2 当前阶段成功标准

后台 V1 完成后，应至少满足：

- 测试人员可在页面内调整游客初始积分、新用户积分、基础评测扣分、维度解锁扣分
- 运营可在页面内调整手机号评测模块配置、客服配置、合规开关
- 客服可搜索用户、查看积分流水、手动加减积分
- 审核人员可查看并处理充值订单
- 排障人员可查看使用记录，定位评测失败、积分扣减、解锁问题

## 4. 用户角色

当前阶段建议先定义 4 类后台角色。

### 4.1 超级管理员

能力范围：

- 查看并修改全部系统配置
- 查看全部用户与记录
- 调整用户积分
- 审核充值订单
- 开启 / 关闭模块与页面
- 管理渠道级配置

### 4.2 运营管理员

能力范围：

- 修改运行时配置
- 调整积分规则、客服配置、渠道配置
- 查看用户与评测记录
- 不负责高风险资金类审批时，可不开放充值审核

### 4.3 客服 / 审单人员

能力范围：

- 搜索用户
- 查看用户详情、积分流水、评测记录
- 处理充值订单审核
- 可按规则手动加积分

### 4.4 只读观察者

能力范围：

- 仅可查看数据，不可修改配置，不可调整积分

补充说明：

- 当前后端只有 `X-Internal-Admin-Token` 的单 token 能力
- 因此角色系统在 V1 可以先体现在产品设计中，但真正细粒度权限控制可以放到 V1.5 / V2

## 5. 核心使用场景

### 5.1 本地测试快速调参

典型场景：

- 把游客初始积分从 `100` 改为 `1000`
- 把基础评测扣分从 `100` 改成 `0` 方便联调
- 把维度解锁扣分从 `50` 改成 `0` 方便前端验收
- 把免费维度从 `3` 个改成更多，测试不同展示状态

### 5.2 运营策略调整

典型场景：

- 调整手机号评测九维度展示顺序
- 指定某个维度免费解锁
- 修改客服微信 / 联系说明
- 调整渠道级隐藏模块或安全模式

### 5.3 用户问题排查

典型场景：

- 用户反馈“积分不够 / 被多扣”
- 用户反馈“充值后没到账”
- 用户反馈“评测失败 / 页面卡住”
- 客服需要查看用户最近评测记录和积分流水

### 5.4 审核与风控

典型场景：

- 查看待审核充值订单
- 审核通过后发放积分
- 审核拒绝并记录原因
- 快速查看某渠道当前是否处于安全模式

## 6. 产品边界

### 6.1 当前明确纳入范围

- 运行时配置管理
- 手机号评测模块配置
- 用户与积分管理
- 充值订单审核
- 使用记录排查
- 合规与渠道开关
- 客服与私域联系配置

### 6.2 当前明确不纳入范围

- 完整 BI 数据分析平台
- 完整财务结算系统
- 面向外部代理商的独立后台
- 富文本内容 CMS
- 手机端后台适配优先
- 复杂工作流审批链

## 7. 信息架构

后台 V1 建议采用左侧导航结构，桌面端优先。

一级导航建议如下：

1. 工作台
2. 运行时配置
3. 手机号评测配置
4. 用户中心
5. 积分中心
6. 充值订单
7. 评测与使用记录
8. 合规与渠道
9. 客服与私域
10. 系统与审计

说明：

- 如果 V1 人力有限，可将 `积分中心` 合并进 `用户中心`
- `系统与审计` 可以先只保留 token 登录入口和配置变更记录占位

## 8. 功能需求

### 8.1 后台登录与访问控制

#### 目标

先把“能进入后台”这件事做顺，再逐步补全账号权限体系。

#### V1 要求

- 提供后台登录页
- 登录方式先支持输入 `Internal Admin Token`
- 登录后本地安全存储 token，并自动为请求附带 `X-Internal-Admin-Token`
- 支持退出登录
- token 失效时自动跳回登录页

#### V1.5 / V2 方向

- 独立后台管理员账户
- 角色与权限模型
- 操作人身份追踪

### 8.2 工作台

#### 目标

让管理员打开后台后第一眼看到最重要的信息和最常用的动作。

#### V1 建议内容

- 今日评测次数
- 今日完成评测次数
- 今日失败评测次数
- 当前用户总数
- 当前游客数 / 注册用户数
- 待审核充值订单数
- 当前是否处于安全模式
- 当前基础评测扣分 / 维度解锁扣分

#### V1 快捷动作

- 一键进入“游客初始积分配置”
- 一键进入“手机号评测扣分配置”
- 一键进入“待审核充值订单”
- 一键进入“用户查询”

#### 本地测试快捷面板

这是 V1 的重点能力，建议单独展示：

- 将游客初始积分快速设为 `100 / 500 / 1000`
- 将基础评测扣分快速设为 `0 / 10 / 100`
- 将维度解锁扣分快速设为 `0 / 10 / 50`
- 一键恢复当前环境默认值
- 预览“当前前台公开配置”快照

### 8.3 运行时配置中心

#### 目标

把目前零散的环境变量思路，统一成页面化配置。

#### 范围

按 `scope_type + scope_key` 管理配置：

- 全局配置：`global/default`
- 渠道配置：`channel/<channel_key>`

#### V1 需要支持的配置项

##### 积分相关

- `points.initial_grant`
- `points.guest_initial_grant`

##### 手机号评测相关

- `phone_review.base_points_cost`
- `phone_review.aspect_unlock_points_cost`
- `phone_review.free_aspect_keys`
- `phone_review.aspect_order`
- `phone_review.unlock_enforcement_enabled`

##### 充值相关

- `recharge.packages`

##### 客服相关

- `customer_service.contact_url`
- `customer_service.qr_code_url`
- `customer_service.guidance_text`

##### 合规相关

- `compliance.safe_mode_enabled`
- `compliance.safe_modules`
- `compliance.hidden_modules`
- `compliance.hidden_pages`

#### 交互要求

- 表单化编辑，不要求运营人员直接写 JSON
- 保存前展示“变更前 / 变更后”
- 高风险配置需要二次确认
- 支持查看最后更新时间

### 8.4 手机号评测配置页

#### 目标

把最核心商业模块的规则独立出来，不埋在通用配置页里。

#### V1 需要支持

- 是否启用手机号评测模块
- 基础评测扣分
- 单维度解锁扣分
- 免费维度选择
- 九维度展示顺序调整
- 是否开启解锁机制

#### V1.5 方向

- 长期使用建议区的运营强化开关
- 特定维度文案策略模板切换
- 面向不同渠道的评测配置模板

### 8.5 用户中心

#### 目标

承接“查用户、看状态、处理问题”的核心场景。

#### 用户列表

V1 需要支持：

- 关键词搜索
- 查看用户类型：游客 / 注册用户
- 查看昵称、创建时间、最近活跃时间
- 查看当前积分余额、冻结积分

#### 用户详情

V1 需要展示：

- `user_id`
- 用户状态
- 昵称 / 头像
- 创建时间 / 更新时间 / 最近活跃时间
- openid / unionid（如有）
- guest_channel / guest_key（如有）
- 当前积分余额 / 冻结积分

#### 用户详情页动作

- 查看积分流水
- 手动加积分
- 手动扣积分
- 查看该用户相关评测记录
- 查看该用户相关充值订单

说明：

- 当前后端已经有用户列表、用户详情、积分流水、积分调整能力
- 但“后台查看该用户全部评测记录”仍建议补充专用内部接口

### 8.6 积分中心

#### 目标

把积分体系从“用户字段”升级为可管理的运营资产。

#### V1 重点

- 按用户调整积分
- 查看积分流水
- 按原因记录备注
- 明确区分“发放”与“扣减”

#### 调整积分时要求

- 必填调整原因
- 自动生成操作流水
- 操作完成后立即返回最新余额
- 对余额不足的扣减给出明确错误提示

#### V1.5 方向

- 批量发积分
- 批量扣积分
- 任务奖励发放
- 活动积分投放

### 8.7 充值订单管理

#### 目标

让客服 / 审单人员可以在后台内完成“看单 - 审单 - 发放积分”闭环。

#### 列表页

V1 需要支持：

- 按状态筛选：待审核 / 已通过 / 已拒绝
- 按用户筛选
- 按来源筛选
- 按渠道筛选

#### 详情页

V1 需要展示：

- 订单号
- 用户信息
- 套餐信息
- 金额
- 应发积分
- 证明图片 / 外部订单号
- 备注
- 审核状态

#### 审核动作

- 通过
- 拒绝
- 填写审核备注
- 审核完成后返回积分发放结果

### 8.8 评测与使用记录

#### 目标

解决“出了问题不知道去哪查”的现状。

#### 使用记录

当前后端已有 `usage_records`，V1 应支持：

- 按用户筛选
- 按场景筛选
- 按状态筛选
- 查看请求摘要
- 查看结果摘要

#### 评测记录

这是后台 V1 非常需要但当前后端尚未完整提供的能力。

V1 应支持：

- 查看全站手机号评测记录
- 按手机号、用户、状态筛选
- 查看评测状态：处理中 / 完成 / 失败
- 查看综合评分
- 查看创建时间
- 查看失败原因
- 进入单条详情页查看完整结果结构

说明：

- 当前用户端已有 `reviews` 相关接口
- 但后台还缺“内部全量评测记录查询接口”
- 该能力应作为后台开发的新增后端需求

### 8.9 合规与渠道控制

#### 目标

让后台能在风险场景下快速切换产品形态。

#### V1 需要支持

- 安全模式总开关
- 安全模块白名单配置
- 隐藏模块配置
- 隐藏页面配置
- 按渠道覆盖配置

#### 关键要求

- 紧急操作必须足够快
- 页面上明确显示“当前全局状态”和“当前渠道状态”
- 配置生效后，前台应立即按公开配置变化

### 8.10 客服与私域配置

#### 目标

把当前写死在前端的客服方式改成后台可维护。

#### V1 需要支持

- 客服联系方式
- 客服二维码图片地址
- 引导说明文案

#### V1.5 方向

- 按页面场景配置不同客服文案
- 按渠道配置不同客服入口
- 标记用户是否已进入私域

### 8.11 系统与审计

#### V1 最小要求

- 后台登录入口
- 当前生效中的内部 token 说明
- 关键操作确认

#### V1.5 / V2 要求

- 操作日志
- 配置变更日志
- 积分调整日志按操作人追踪
- 充值审核日志按操作人追踪

## 9. 数据真相与接口对齐

### 9.1 当前后端可直接复用的能力

以下内部接口已经存在，可优先复用：

- `GET /api/v1/internal/runtime-config`
- `PUT /api/v1/internal/runtime-config`
- `GET /api/v1/internal/users`
- `GET /api/v1/internal/users/{user_id}`
- `GET /api/v1/internal/users/{user_id}/points/ledger`
- `POST /api/v1/internal/users/{user_id}/points/adjust`
- `GET /api/v1/internal/usage-records`
- `GET /api/v1/internal/recharge-orders`
- `GET /api/v1/internal/recharge-orders/{order_id}`
- `POST /api/v1/internal/recharge-orders/{order_id}/review`

### 9.2 当前后端建议新增的能力

为了让后台 V1 真正可用，建议新增：

1. 后台登录态接口  
   - 当前只有 header token 校验，建议至少增加“后台登录页 + token 校验接口”体验层支持

2. 工作台汇总接口  
   - 返回今日评测量、失败量、待审核订单数、用户总数等概览数据

3. 内部评测记录列表与详情接口  
   - 面向后台查看全量手机号评测记录

4. 配置项元数据接口（可选）  
   - 告诉前端哪些配置项是数字、布尔、数组、JSON

5. 操作日志接口  
   - 用于后续审计

6. 推广大使与任务配置接口  
   - 当前产品 PRD 已有要求，但后端基础能力尚未补齐

### 9.3 V1 后台前端可以接受的现实

如果希望最快落地，后台前端 V1 可以接受以下折中：

- 配置项表单结构先由前端写死
- 权限先统一使用单一 internal token
- 审计日志先不做完整版本
- 统计看板先做轻量版

## 10. 交互与安全要求

### 10.1 交互原则

- 桌面端优先，推荐 `1280px+` 使用体验
- 配置页先强调“清楚、稳定、少误操作”，不追求复杂炫技
- 高风险动作必须二次确认

### 10.2 高风险动作

以下操作应视为高风险：

- 扣减用户积分
- 修改游客 / 新用户初始积分
- 修改手机号评测扣分规则
- 关闭核心模块
- 开启安全模式
- 审核通过充值订单

### 10.3 容错要求

- 保存失败时要展示明确原因
- 配置变更应尽量原子化
- 审核接口要避免重复操作
- 页面刷新后应能恢复当前筛选条件

## 11. 阶段实施建议

### 11.1 Phase A：测试配置台

优先级最高，必须先做。

包含：

- 后台登录
- 运行时配置中心
- 手机号评测配置页
- 本地测试快捷面板

目标：

- 让测试人员不再改环境变量或敲接口

### 11.2 Phase B：运营工作台

第二优先级。

包含：

- 工作台首页
- 用户中心
- 积分中心
- 充值订单管理
- 客服配置

目标：

- 让客服与运营可以独立完成日常支持工作

### 11.3 Phase C：完整运营后台

后续阶段。

包含：

- 全站评测记录管理
- 更完整的渠道管理
- 推广大使配置
- 任务积分配置
- 审计与角色系统

## 12. 成功指标

后台上线后，希望明显改善以下指标：

- 修改测试积分规则所需时间
- 查询单个用户问题所需时间
- 处理一笔充值订单所需时间
- 从发现风险到切换安全模式所需时间
- 测试阶段因“配置不便”导致的阻塞次数

## 13. 待确认事项

- 后台是否单独部署，还是先挂在同一服务下
- 后台前端技术栈是否沿用当前前端体系
- V1 是否只允许单 token 进入
- 是否需要“批量发积分”能力进入 V1
- 是否需要“创建测试用户 / 测试订单”能力进入 V1
- 评测记录后台页是否要求查看完整九宫盘内容

## 14. 文档治理规则

- 本文档是后台管理系统的产品初稿，不替代主产品 PRD
- 凡是后台能力影响前台业务规则的，必须与 `product-prd.md` 保持一致
- 若后台功能范围变化，应优先更新本文档，再进入实现
