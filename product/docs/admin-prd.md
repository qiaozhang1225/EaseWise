## 1. 背景与问题

随着 `易如反掌` 逐步进入正式上线和持续运营阶段，系统需要一套完整、系统化的管理后台，承接上线后的日常管理、运营配置、问题排查与业务协同。

最直接的问题包括：

- 关键业务配置缺少统一、可视化的管理入口
- 用户、订单、收益、推广合作等核心业务缺少集中管理与排查能力
- 运营、管理、开发协同所需的数据查看和操作入口分散
- 部分内部能力虽然已经存在，但缺少适合日常运营场景的后台承接

因此，后台管理系统的第一目标不是“做一个完整企业后台”，而是：

> 先做一套面向正式运营阶段的系统化管理后台，兼顾开发、管理和运营所需的查看、配置、审核与排查能力。

## 2. 产品定位

`易如反掌后台` 是服务于整个系统的桌面端管理后台，面向开发者、管理者和运营者，承接正式上线后的日常管理与运营协同。

它不是面向普通用户的产品能力，而是产品系统的管理控制台。

当前阶段的定位可以压缩为一句话：

> 用统一的后台页面，承接系统上线后的运营管理、配置调整、订单处理、用户排查和推广合作管理。

## 3. 当前阶段目标

### 3.1 V1 目标

V1 先解决以下四类问题：

1. 能快速改配置  
2. 能快速查用户  
3. 能快速调积分  
4. 能快速处理充值审核、提现审核与问题排查  

### 3.2 当前阶段成功标准

后台 V1 完成后，应至少满足：

- 管理者可在页面内调整过渡期游客初始积分、新用户积分、基础评测扣分、维度解锁扣分
- 运营可在页面内调整手机号评测模块配置、客服配置、合规开关
- 客服可搜索用户、查看积分流水、手动加减积分
- 审核人员可查看并处理充值订单
- 审核人员可查看并处理提现申请
- 排障人员可查看使用记录，定位评测失败、积分扣减、解锁问题

## 4. 用户角色

定义 4 类后台角色，作为后续权限控制与页面按钮权限的基础。

补充说明：

- V1 统一使用单一 `Internal Admin Token` 进入后台
- 角色权限体现在页面可见性、按钮可操作性和数据范围控制上
- 细粒度的账号体系和审计追踪放到后续版本

### 4.1 超级管理员

能力范围：

- 查看并修改全部系统配置
- 查看全部用户与记录
- 调整用户积分
- 审核充值订单
- 开启 / 关闭模块与页面
- 管理渠道级配置
- 审核提现申请
- 审核推广大使申请
- 查看并处理返佣记录
- 查看并处理退款流程
- 管理角色与权限配置

补充说明：

- 该角色拥有后台全部一级导航的查看和操作权限
- 该角色可以执行所有高风险操作
- 该角色是所有其他角色权限上限的集合

### 4.2 运营管理员

能力范围：

- 查看数据大盘、订单管理、用户管理、功能管理、推广合作、系统配置
- 调整积分规则、客服配置、渠道配置
- 调整手机号评测、智能体、黄历等功能配置
- 查看用户与评测记录
- 查看返佣记录与提现申请
- 处理非资金敏感类的日常运营配置

默认不包含：

- 角色与权限配置
- 高风险资金类审批
- 退款执行

补充说明：

- 如果需要，运营管理员可在权限上被额外授权处理部分推广合作审核
- 该角色更偏向配置、查看和运营分析，不应承担最终资金责任

### 4.3 客服 / 审单人员

能力范围：

- 搜索用户
- 查看用户详情、积分流水、评测记录
- 处理充值订单审核
- 可按规则手动加积分
- 查看提现申请与审核状态
- 处理提现审核与打款失败后的人工跟进
- 查看返佣记录用于问题排查

默认不包含：

- 系统配置修改
- 功能开关修改
- 角色权限修改
- 推广规则修改

补充说明：

- 该角色更偏向一线处理与问题排查
- 涉及资金流转、退款和提现时，应保留完整操作记录

### 4.4 只读观察者

能力范围：

- 仅可查看数据，不可修改配置，不可调整积分
- 可查看数据大盘、订单、用户、功能配置、推广合作记录和系统配置
- 不可执行任何审核、调整、开关变更或提现相关操作

补充说明：

- 该角色适合只读查看、汇报、复盘和排障协助
- 只读权限不应接触高风险操作入口

### 4.5 权限原则

- 所有角色均应支持按模块控制按钮显隐
- 所有高风险操作都应要求二次确认
- 金额、积分、提现、退款、开关关闭等动作应默认高风险
- 若后续引入正式账号体系，应保留角色与操作日志的扩展位

## 5. 核心使用场景

### 5.1 配置快速调整

典型场景：

- 把过渡期游客初始积分从 `100` 改为 `1000`
- 把基础评测扣分从 `100` 改成 `0`
- 把维度解锁扣分从 `50` 改成 `0`
- 把免费维度从 `3` 个改成更多

### 5.2 运营策略调整

典型场景：

- 调整手机号评测九维度展示顺序
- 指定某个维度免费解锁
- 修改客服微信 / 联系说明
- 调整渠道级隐藏模块或安全模式

### 5.3 用户问题排查

典型场景：

- 用户反馈“积分不够 / 被多扣”
- 用户反馈“充值后没到账”
- 用户反馈“评测失败 / 页面卡住”
- 客服需要查看用户最近评测记录和积分流水

### 5.4 审核与风控

典型场景：

- 查看待审核充值订单
- 审核通过后发放积分
- 审核拒绝并记录原因
- 快速查看某渠道当前是否处于安全模式

## 7. 信息架构

后台 V1 采用“左侧树状导航 + 右侧主内容区”的常规桌面端管理后台布局，桌面端优先。

### 7.1 一级导航

采用 6 个一级导航：

1. 数据大盘
2. 订单管理
3. 用户管理
4. 功能管理
5. 推广合作
6. 系统配置

说明：

- 首页默认进入 `数据大盘`
- `数据大盘`、`订单管理`、`用户管理`、`系统配置` 均使用单页面承接
- `推广合作` 保留二级导航，承接审核、返佣和提现等相对独立的业务流程
- `功能管理` 保留二级导航，当前至少包含 `黄历相关设置` 与 `数字奇门手机号评测`
- `功能管理` 作为独立一级模块，用于承接业务功能本身的开关、规则、记录与排查，不与平台级设置完全混用

### 7.2 数据大盘

`数据大盘` 不设置二级导航，只保留一个首页级页面，用于展示整个系统最核心的经营与运行数据。

数据大盘不堆叠过细的业务列表，而以摘要、趋势、核心指标和快速跳转为主。页面分为以下 4 个区域：

#### 1. 收益区域

这是数据大盘中优先级最高的区域，也是页面的核心内容。

关键指标：

- 昨日收益
- 本月收益
- 上月收益
- 当年收益
- 上一年收益
- 累计总收益
- 已产生返佣金额
- 已提现金额
- 净收益

每个收益卡片或收益板块补充以下细节：

- 对应周期内的支付订单数
- 对应周期内的平均客单价
- 简单趋势图或环比走势
- 退款影响金额（如该周期存在退款）

#### 2. 用户区域

用于展示系统整体用户状况。

关键指标：

- 当前总用户量
- 今日新增用户
- 活跃用户指标，例如 `DAU`、`MAU`
- 推广大使数量
- 高级推广大使数量
- 由推广带来的新增用户数量

如页面空间允许，可补充：

- 用户增长趋势图
- 渠道分布摘要
- 用户活跃趋势摘要

#### 3. 订单区域

用于展示订单系统的关键运行状态。

关键指标：

- 今日支付订单数
- 今日支付金额
- 待完成订单数
- 退款中订单数
- 已退款金额
- 已关闭订单数

如页面空间允许，可补充：

- 订单状态分布
- 近 7 日订单趋势
- 异常订单提醒或退款风险提醒

#### 4. 推广合作区域

用于展示推广合作相关的关键摘要。

关键指标：

- 待审核推广申请数
- 待审核提现申请数
- 待结算返佣金额
- 已产生返佣金额
- 已撤回返佣金额
- 已提现金额

如页面空间允许，可补充：

- 推广带来的新增用户数
- 推广带来的支付订单数
- 推广渠道转化摘要

### 7.3 订单管理

`订单管理` 不拆二级导航，使用一个主页面承接。

页面形态以表格为主，并支持以下能力：

- 搜索与筛选
- 查看订单详情
- 执行订单相关操作

订单管理页面至少支持：

- 按订单状态搜索
- 按用户名或用户标识搜索
- 按金额范围搜索
- 按时间范围搜索

列表页展示的基础字段：

- 订单 ID
- 用户信息
- 订单金额
- 订单状态
- 订单来源 / 渠道
- 创建时间
- 支付时间
- 完成时间

订单详情页应承接更完整的订单信息、状态流转记录以及退款处理等操作入口。

### 7.4 用户管理

`用户管理` 不拆二级导航，使用一个主页面承接。

页面形态以表格为主，重点是帮助管理者快速搜索并定位特定用户。

用户管理页面支持：

- 全面搜索用户
- 查看用户详情
- 查看用户使用记录
- 管理积分
- 查看推广资金资产
- 启用或禁用用户
- 重新设置用户身份

列表页展示的基础字段：

- 用户 ID
- 昵称 / 姓名 / 手机号
- 当前用户状态
- 积分余额
- 可提现余额
- 冻结中返佣余额
- 当前身份等级
- 注册渠道
- 创建时间
- 最近活跃时间

定位到具体用户后，后台应至少支持以下基础操作：

1. 查看使用记录
2. 积分管理：增加、减少或重新设置积分
3. 状态管理：禁用或启用用户
4. 身份设置：重新设置用户身份，例如普通用户、推广大使、高级推广大使

### 7.5 功能管理

`功能管理` 保留二级导航，当前先拆为两个页面：

1. 黄历相关设置
2. 数字奇门手机号评测

该模块主要承接业务功能本身的管理，不是做通用系统设置，而是围绕具体功能入口、功能规则和功能可见性展开。

纳入的内容包括：

- 按渠道总开关
- 单功能开关
- 手机号评测相关规则
- 智能体玄学技能开关
- 功能入口显隐控制
- 功能使用记录与搜索

页面内按功能分组展示：

- 黄历相关设置
- 数字奇门手机号评测
- 智能体玄学技能调用记录
- 其他后续业务能力

其中，`黄历相关设置` 页面 V1 可先作为占位页面，保留后续开发任务入口。`数字奇门手机号评测` 页面是当前功能管理的主页面，承接手机号评测规则、维度解锁规则、功能开关和功能使用记录。

其中需要重点承接的规则包括：

- 手机号评测基础积分消耗
- 手机号评测九个维度的解锁积分规则
- 智能体相关技能的可用状态
- 历史使用记录的显隐规则
- 功能使用记录搜索、详情抽屉、关联用户与订单跳转

### 7.6 推广合作

`推广合作` 是独立一级模块，保留二级导航，具体包括：

1. 推广审核
2. 提现申请审核
3. 返佣记录
4. 规则配置

各二级页面定位如下：

- `推广审核`
  - 主要承接用户申请成为 `推广大使` 或 `高级推广大使` 的审核流程
  - 页面以表格为主，支持查看申请详情、通过和拒绝
- `提现申请审核`
  - 承接推广用户的提现申请审核
  - 页面以表格为主，支持查看申请详情、审核通过、拒绝、自动打款状态查看与后续处理
- `返佣记录`
  - 支持按用户、大使、订单等条件搜索返佣流水
  - 重点解决“某用户是否产生返佣、返佣为何未到账、返佣是否被撤回”等排查需求
- `规则配置`
  - 承接所有推广合作相关配置
  - 例如推广门槛、返佣比例、最低提现门槛、订单完成判定天数等

说明：

- `推广概览` 不单独放在本模块内，而统一纳入 `数据大盘`
- 推广相关的运营规则放在本模块的 `规则配置` 中，不放入通用系统配置页

### 7.7 系统配置

`系统配置` 不拆二级导航，使用一个主页面承接。

该页面承接平台级、全局性的关键设置，包括：

1. 充值套餐配置
2. 积分规则配置
3. 功能开关配置
4. 客服联系方式配置
5. 大模型密钥管理

各配置分组的定位如下：

- `充值套餐配置`
  - 管理充值金额与积分对应关系
  - 支持新增套餐、删除套餐、启用或停用套餐
- `积分规则配置`
  - 管理初始积分、基础积分规则以及全局积分策略
- `功能开关配置`
  - 管理系统层级的重要开关项
  - 与 `功能管理` 配合使用，前者偏全局控制，后者偏具体功能规则
- `客服联系方式配置`
  - 管理客服图片、二维码、联系方式文案等内容
- `大模型密钥管理`
  - 当前作为占位配置分组
  - 仅展示脱敏密钥配置与未来扩展入口
  - V1 不影响线上 DeepSeek 调用，不暴露真实环境变量密钥

补充说明：

- 推广规则配置不放在 `系统配置` 中，而归入 `推广合作 > 规则配置`
- 先保证“能管、能查、能配”，不要求把所有设置做成复杂工作流

## 8. 功能需求

### 8.1 后台登录与访问控制

#### 目标

先把“能进入后台”这件事做顺，再逐步补全账号权限体系。

#### V1 要求

- 提供后台登录页
- 登录方式先支持输入 `Internal Admin Token`
- 登录后本地安全存储 token，并自动为请求附带 `X-Internal-Admin-Token`
- 支持退出登录
- token 失效时自动跳回登录页

#### V1.5 / V2 方向

- 独立后台管理员账户
- 角色与权限模型
- 操作人身份追踪

### 8.2 数据大盘

#### 目标

让管理员进入后台后，第一眼看到系统最核心的经营数据与运行状态。

#### 核心口径

- `收益` 口径以用户 `已支付金额` 为准
- `已完成订单` 主要服务于推广返佣结算，不直接作为收益认定口径
- `已产生返佣金额`、`已提现金额` 作为独立指标展示
- `净收益` 建议暂按 `已支付金额 - 已产生返佣金额` 展示，`已提现金额` 单独列示，避免重复扣减
- `退款金额` 单独展示，用于反映退款风险与资金回流情况

#### 页面结构

页面分为四个区域：

1. 收益区域
2. 用户区域
3. 订单区域
4. 推广合作区域

#### 收益区域

展示：

- 昨日收益
- 本月收益
- 上月收益
- 当年收益
- 上一年收益
- 累计总收益
- 已产生返佣金额
- 已提现金额
- 净收益

补充：

- 对应周期订单数
- 平均客单价
- 趋势图
- 退款影响摘要

#### 用户区域

展示：

- 当前总用户量
- 今日新增用户
- 活跃用户指标，如 `DAU`、`MAU`
- 推广大使数量
- 高级推广大使数量
- 推广新增用户数

#### 订单区域

展示：

- 今日支付订单数
- 今日支付金额
- 待完成订单数
- 退款中订单数
- 已退款金额
- 已关闭订单数

#### 推广合作区域

展示：

- 待审核推广申请数

补充说明：

- 若订单在观察期内未出现积分消费记录，则支付满观察期后自动判定为 `已完成`
- 默认观察期按 `7 天` 执行，但该数值必须支持后台配置
- `已完成` 主要用于返佣结算，不作为收益统计唯一口径

- 查看关联返佣记录

- 发起退款流程

#### 退款处理要求

退款不由用户在系统内自助完成，而由用户先提交退款诉求，再由管理员在后台审核并执行。

规则如下：

- 只有未产生积分消费的充值订单才允许退款
- 一旦用户在充值后已经产生积分消费，则该订单不允许退款
- 退款审核通过后，系统进入退款流程，并同步将订单状态更新为 `退款中`
- 退款成功后，系统必须同步变更订单状态、冲减退款相关统计，并撤回上级返佣金额
- 返佣撤回优先冲减冻结中返佣余额；已结算部分从可提现余额中冲减
- 用户已获得的积分当前不要求自动扣回
- 退款失败时，后台必须保留失败原因，并支持后续重试或人工介入

#### 容错要求

- 同一订单的退款操作必须避免重复提交
- 涉及资金相关的高风险操作必须二次确认
- 所有退款处理动作必须保留操作记录

- 四柱八字
- 其他玄学分析能力

- 智能体基础聊天能力
- “我的”页面相关基础功能

- 手机号评测开关

- 其他后续玄学功能开关

补充说明：

- 智能体开关不是关闭整个智能体，而是关闭其玄学技能调用能力
- 当功能关闭后，前台应隐藏对应功能入口
- 当功能关闭后，相关历史使用记录也应一并隐藏

#### 功能分组

- 高风险配置修改必须二次确认
- 必须展示规则快照，避免历史订单直接套用当前配置

### 8.7 系统配置

#### 目标

承接平台级、全局性的基础设置，不承接具体业务功能细则，也不承接推广合作专属规则。

#### 页面要求

系统配置使用单页面结构，不拆二级页面。

#### V1 需要支持

- 充值套餐配置
- 积分基础规则配置
- 少量平台级开关配置
- 客服联系方式配置
- 大模型密钥管理占位

#### 配置分组

- 充值套餐配置
- 积分规则配置
- 平台级开关配置
- 客服联系方式配置
- 大模型密钥管理

#### 充值套餐配置

至少需要支持：

- 新增套餐
- 删除套餐
- 启用 / 停用套餐
- 配置金额与积分对应关系

每个套餐至少包含：

- 套餐标识
- 套餐名称
- 金额
- 对应积分数
- 启用状态
- 排序

补充说明：

- 如全部套餐被删除或停用，则视为系统当前不允许充值

#### 积分规则配置

至少支持：

- 过渡期游客初始积分
- 注册用户初始积分
- 全局积分基础规则

补充说明：

- 过渡期游客初始积分仅用于手机号短信验证或微信 UnionID 登录上线前的兼容，后续应停用或调整为 0
- 业务功能的具体消耗规则统一放在 `功能管理`
- 推广返佣资金相关规则统一放在 `推广合作 > 规则配置`

#### 平台级开关配置

该区域只承接少量平台级开关项，例如：

- 充值入口是否开放
- 某些全局性展示开关

补充说明：

- 具体业务功能的开关与规则，统一在 `功能管理` 页面维护
- 推广合作相关规则，不放在 `系统配置` 中维护

#### 客服联系方式配置

至少支持：

- 客服联系方式文本
- 客服二维码图片
- 引导说明文案

#### 大模型密钥管理

`大模型密钥管理` 当前作为系统配置页内的独立配置分组，V1 只做占位和数据结构预留，不接入真实 DeepSeek 调用链。

页面能力：

- 展示密钥列表占位表格
- 字段包括密钥名称、模型供应商、模型名称、启用状态、备注、最后更新时间、最后操作人
- 密钥值默认脱敏展示，例如 `sk-****abcd`
- 页面提供 `新增密钥`、`编辑`、`启用/停用`、`删除` 按钮
- V1 需明确标注“当前为占位配置，不影响线上 DeepSeek 调用”

后端预留：

- 新增 `llm_api_keys` 数据表或等价配置模型
- 字段包括 `id`、`provider`、`model`、`display_name`、`masked_key`、`secret_ref`、`enabled`、`priority`、`remark`、`created_at`、`updated_at`
- 不在数据库明文保存真实 key
- 如果 V1 暂无密钥加密或密钥托管方案，真实 key 仍走环境变量，后台只展示占位配置

后续扩展：

- 并发监控
- 失败率统计
- 自动切换 key
- 按功能模块选择不同模型 key

验收标准：

- 系统配置页能看到 `大模型密钥管理` 分组
- 页面明确提示当前为占位，不影响线上 DeepSeek 调用
- 不泄露真实 `DEEPSEEK_API_KEY`

#### 通用交互要求

- 配置项应按表单方式维护，不要求直接编辑底层 JSON
- 保存前应展示变更前后内容
- 页面中需明确标注配置作用范围
- 高风险配置必须二次确认
- 支持查看最后更新时间与最后操作人

#### 说明

- 推广规则配置不放在 `系统配置` 中，而归入 `推广合作 > 规则配置`
- 具体业务功能规则不放在 `系统配置` 中，而归入 `功能管理`
- 当前先保证“能管、能查、能配”

## 9. 数据真相与接口对齐

### 9.1 数据口径

- 后台统计默认按系统时区和自然日 / 自然月 / 自然年计算
- `收益` 口径以用户 `已支付金额` 为准
- `已完成订单` 主要用于推广返佣结算，不作为收益认定口径
- `净收益` 建议暂按 `已支付金额 - 已产生返佣金额` 展示
- `已提现金额`、`已退款金额` 建议作为独立指标展示
- 退款成功后应同步影响订单状态、退款统计和返佣撤回

### 9.2 当前可直接复用的接口能力

#### 后台访问控制

- 当前仅有 `X-Internal-Admin-Token` 的 header 校验
- 暂无独立后台登录接口，后台前端需要先以 token 登录页承接

#### 数据大盘参考数据源

- `GET /api/v1/internal/users`
- `GET /api/v1/internal/billing/recharge-orders`
- `GET /api/v1/internal/platform/usage-records`
- `GET /api/v1/internal/runtime-config`
- 现有接口可作为统计来源，但大盘补专用汇总接口

#### 订单管理

- `GET /api/v1/internal/billing/recharge-orders`
- `GET /api/v1/internal/billing/recharge-orders/{order_id}`
- `POST /api/v1/internal/billing/recharge-orders/{order_id}/review`

说明：

- 现有接口已覆盖订单列表、详情和基础审核
- 退款全流程、退款失败处理、人工重试等能力仍需补充

#### 用户管理

- `GET /api/v1/internal/users`
- `GET /api/v1/internal/users/{user_id}`
- `GET /api/v1/internal/users/{user_id}/points/ledger`
- `POST /api/v1/internal/users/{user_id}/points/adjust`
- `GET /api/v1/internal/platform/usage-records`

说明：

- 现有接口已覆盖用户列表、详情、积分流水、手动调积分和使用记录查询
- 用户封禁 / 启用、身份重置等管理动作仍需新增接口

#### 功能管理 / 系统配置

- `GET /api/v1/internal/runtime-config`
- `PUT /api/v1/internal/runtime-config`
- `GET /api/v1/runtime-config/public`

说明：

- 当前已支持通用配置的读取和写入
- 仍缺少配置元数据、表单类型说明、操作日志和变更差异展示相关接口

#### 用户侧评测能力（后台排查参考）

- `POST /api/v1/phone-qimen/reviews`
- `GET /api/v1/phone-qimen/reviews`
- `GET /api/v1/phone-qimen/reviews/{review_id}`
- `GET /api/v1/phone-qimen/reviews/{review_id}/aspect-unlocks`
- `POST /api/v1/phone-qimen/reviews/{review_id}/aspect-unlocks`

说明：

- 这组接口主要服务前台用户
- 后台全量评测记录查询使用 `/api/v1/internal/phone-qimen/reviews`

### 9.3 当前建议新增的接口

为了让后台 V1 真正可用，新增：

1. 数据大盘汇总接口  
   - 返回收益、用户、订单、推广合作等核心摘要数据

2. 内部评测记录列表与详情接口  
   - 面向后台查看全量手机号评测记录

3. 用户状态与身份管理接口  
   - 支持禁用 / 启用、身份重置、上级关系调整

4. 推广审核接口  
   - 支持申请列表、申请详情、通过、拒绝

5. 提现申请接口  
   - 支持申请列表、申请详情、审核、打款失败重试、手动打款完成

6. 返佣记录接口  
   - 支持按用户、订单、大使、状态、时间范围查询

7. 推广规则接口  
   - 支持查询和保存推广门槛、返佣比例、最低提现门槛、订单完成判定天数等规则

8. 退款处理接口  
   - 支持订单退款申请、审核、执行、失败重试

9. 配置项元数据接口

10. 操作日志接口  
    - 用于后续审计和高风险操作追踪

### 9.4 V1 后台前端可以接受的现实

后台前端 V1 可以接受以下折中：

- 配置项表单结构先由前端写死
- 权限先统一使用单一 internal token
- 审计日志先不做完整版本
- 统计看板先做轻量版

## 10. 交互与安全要求

### 10.1 交互原则

- 桌面端优先，推荐 `1280px+` 使用体验
- 后台整体以“清楚、稳定、少误操作”为优先，不追求复杂炫技
- 表格页优先保证搜索、筛选、详情查看和操作效率
- 详情信息优先通过抽屉、侧板、详情层等方式展开，减少页面来回跳转
- 摘要卡片与图表优先支持跳转到对应业务页面
- 配置项应尽量使用表单化维护，不直接暴露底层数据结构
- 所有高风险动作必须二次确认
- 页面应尽量保留当前筛选条件、排序条件和分页状态

### 10.2 高风险动作

以下操作应视为高风险：

- 扣减用户积分
- 调整推广可提现余额或冻结中返佣余额
- 修改过渡期游客 / 新用户初始积分
- 修改手机号评测扣分规则
- 修改推广返佣比例、推广门槛、提现门槛
- 关闭渠道总开关
- 关闭核心功能开关
- 审核通过提现申请
- 发起或执行退款
- 手动撤回返佣
- 手动补发返佣

高风险动作的统一要求：

- 必须二次确认
- 必须展示变更前后内容或操作影响范围
- 必须记录操作时间、操作人和操作原因
- 若涉及金额、积分、提现、退款或返佣，必须保留操作流水
- 若接口返回失败，页面应明确提示失败原因

### 10.3 数据与资金安全

- 所有金额、积分、返佣、提现相关数据应以后端返回为准
- 前端不得以本地缓存或本地推导结果替代真实账务结果
- 同一笔订单、提现、退款、返佣操作应避免重复提交
- 涉及资金流转的动作应具备明确状态流转，不允许出现前端展示成功但后端未落账的情况
- 若出现自动处理失败，应保留失败状态并支持人工介入或重试
- 渠道总开关和核心功能开关的变更应明确作用范围，避免误伤其他渠道

### 10.4 容错要求

- 保存失败时要展示明确原因
- 配置变更应尽量原子化
- 审核接口要避免重复操作
- 页面刷新后应能恢复当前筛选条件
- 长列表页在接口失败时应保留已输入的筛选条件
- 提现、退款、返佣等异步流程在失败时应保留失败状态和失败原因
- 关键配置保存后应支持再次查看当前生效值，避免出现“保存成功但结果不明”

## 11. 阶段实施建议

### 11.1 Phase A：基础管理能力

优先级最高，必须先做。

包含：

- 后台登录
- 数据大盘
- 订单管理
- 用户管理

目标：

- 先建立后台的基础查看与管理能力

### 11.2 Phase B：运营与配置能力

第二优先级。

包含：

- 功能管理
- 推广合作
- 系统配置

目标：

- 让管理者与运营者可以独立完成日常运营配置与业务处理

### 11.3 Phase C：完整运营后台

后续阶段。

包含：

- 全站评测记录管理
- 更完整的渠道管理
- 更完整的推广合作能力
- 审计与角色系统

## 12. 成功指标

后台上线后，希望明显改善以下指标：

- 调整积分规则所需时间
- 查询单个用户问题所需时间
- 处理一笔充值订单所需时间
- 从发现风险到切换安全模式所需时间
- 因“配置不便”导致的阻塞次数

## 13. 待确认事项

当前仍建议保留待确认的内容仅包括：

- 后台是否单独部署，还是先挂在同一服务下
- 后台前端技术栈是否沿用当前前端体系
- V1 是否只允许单 token 进入
- 评测记录后台页是否要求查看完整九宫盘内容
- 是否需要批量发积分能力进入 V1
- 是否需要创建测试用户 / 测试订单能力进入 V1
