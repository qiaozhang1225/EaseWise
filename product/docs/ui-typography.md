# EaseWise UI 字体规范（初稿）

## 目标

本规范用于统一 `EaseWise` 前端中的字体语言，避免页面在不同组件、不同系统、不同开发者手里出现字体角色混乱、数字对不齐、中文气质不统一的问题。

该规范适用于：

- 首页
- 手机号评测页
- 智能体页
- “我的”页
- 弹窗、浮层、导航、卡片、表单

---

## 一、三大字体家族及语义职责

### 1. `font-serif`：文化语义 / 国风主视觉

用于承载“历史、古典、术数、厚重感”的文本内容。

强制使用场景：

- 奇门遁甲相关概念标题
- 天干、地支、门、星、神、宫位、方向名
- 核心批语标题
- 国风品牌标题
- 评分大数字、核心结果标题

设计目标：

- 营造古籍、宣纸、术数体系的文化气质
- 强化权威感和仪式感

禁止事项：

- 不用于长段说明文本
- 不用于普通按钮、输入框、表单标签
- 不用于大段弹窗正文

### 2. `font-sans`：界面说明 / 操作信息

用于承载“现代、清晰、说明性、交互性”的文本内容。

强制使用场景：

- 普通正文
- 输入框提示词
- 操作按钮
- 导航栏
- 弹窗说明
- 标签、表单项、反馈文案

设计目标：

- 保证阅读清晰度
- 保证移动端可读性和一致性

禁止事项：

- 不用于需要国风文化气质的核心术数标题
- 不用于核心评分视觉主数字

### 3. `font-mono`：结构化数据 / 精准对齐

用于承载“计算感、结构化、稳定对齐”的文本内容。

优先使用场景：

- 时间戳
- 编号、流水、客服号
- 账单增减值
- 需要上下或左右严格对齐的数据字段

建议使用场景：

- 天盘地盘中的符号型短文本
- 需要视觉稳定的单字定位位

注意事项：

- `font-mono` 不默认承接所有数字
- 视觉型数字应回归所在语义：品牌/术数结果用 `font-serif`，按钮/说明中的数字用 `font-sans`
- 若内容为中文单字或术数符号，仅使用 `font-mono` 不一定完全解决抖动问题
- 必须同时配合固定容器宽高、居中布局、`leading-none`、`select-none`

## 二、全局字体回退策略

不同系统的默认中文字体差异很大，如果不显式声明回退链，macOS、iOS、Windows、Android 会呈现完全不同的字形气质。

因此必须在全局主题变量中明确声明字体优先级。

推荐定义：

```css
@theme {
  --font-sans: "Inter", "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", ui-sans-serif, system-ui, sans-serif;
  --font-serif: "Noto Serif CJK SC", "Source Han Serif SC", "Songti SC", "Playfair Display", "SimSun", serif;
  --font-mono: "JetBrains Mono", "Sarasa Mono SC", "Fira Code", "Courier New", monospace;
}
```

说明：

- `font-sans` 优先现代 UI 字体，确保按钮、说明、输入内容风格稳定
- `font-serif` 优先中文衬线字体，避免中英混排时气质割裂
- `font-mono` 兼顾英文数字等宽与中文环境下的可读性

## 三、字体角色与业务内容的映射规则

### 必须使用 `font-serif` 的内容

- “奇门遁甲手机号综合测评”
- “立体定盘局象”
- 九宫盘面元素
- 门星神名称
- 方向位名称
- 核心批语标题
- 品牌主标题
- 结果分数主视觉

### 必须使用 `font-sans` 的内容

- 所有按钮文案
- 输入框 placeholder
- 表单标签
- 普通说明文字
- 弹窗正文
- 标签页名称
- 设置项、提示项、空态说明

### 优先使用 `font-mono` 的内容

- 时间
- 流水差额
- 客服联系方式
- 编号 / ID / 调试信息
- 需要严格对齐的数据列
- 聊天气泡时间戳
- 其他需要对齐的数字型字段

### 数字字体的补充规则

- 核心结果分数、积分主数值、仪式感展示数字：优先使用 `font-serif`
- 输入框、按钮、说明文案中的数字：跟随 `font-sans`
- 只有在“对齐稳定性”比“视觉气质”更重要时，才使用 `font-mono`

## 四、字号层级规范

项目内尽量收敛为以下字号层级，避免继续新增零散的 `text-[xx.xpx]`：

| 层级 | 建议字号 | 典型场景 |
| --- | --- | --- |
| Display XL | `32px` | 核心积分、主分数 |
| Display L | `28px` | 大结果数字 |
| Heading Plus | `22px` | 日期、大号小标题、方向位强调 |
| Heading 1 | `20px` | 页头标题、主模块标题 |
| Heading 2 | `17px` | 卡片标题、弹窗标题 |
| Title L | `18px` | 重要次级标题 |
| Title | `15px` | 次级重点标题 |
| Body | `13px` | 主体说明、按钮文字 |
| Caption | `11px` | 辅助说明、表单标签、角标 |
| Micro | `10px` | 极小标签、状态短标识 |
| Tiny | `8.5px` | 极小状态徽标，仅限特殊角标 |

治理原则：

- 优先复用上述层级
- 新增字号必须有明确理由
- `9.5px / 10.5px / 11.5px / 12.5px / 13.5px / 14.5px / 15.5px / 17.5px` 这类散点值后续应逐步收敛

补充说明：

- `60px` 的大水印类字号属于盘面装饰位，不计入常规正文层级，但必须通过专用语义类统一管理

## 五、字重规范

为避免中文界面发黑、发挤，限制字重使用范围：

| 字重 | 使用场景 |
| --- | --- |
| `font-medium` | 次级说明、局部弱强调 |
| `font-semibold` | 小标题、状态标签 |
| `font-bold` | 常规标题、按钮、重点字段 |
| `font-extrabold` | 局部强调标签、短数字强调 |
| `font-black` | 仅限核心评分、强视觉数字、少量术数装饰字 |

禁止事项：

- 中文大段正文禁止使用 `font-black`
- 普通按钮不应滥用 `font-black`
- `font-black` 仅用于需要“视觉锚点”的少数位置

---

## 六、行高与对齐规则

### 1. 大字号衬线标题

当文本满足以下条件时：

- 字号大于等于 `18px`
- 使用 `font-serif`
- 字重大于等于 `font-bold`

必须优先搭配：

- `leading-none`
  或
- `leading-tight`

目的：

- 减少字体自然留白导致的上下空隙
- 保证 `flex items-center` 时视觉真正居中

### 2. 数字与单字定位内容

以下内容必须同时具备：

- 固定宽高容器
- `flex items-center justify-center`
- `leading-none`
- `select-none`

适用场景：

- 引干
- 宫位边缘小标
- 单字数字牌
- 浮层小标签

### 3. 正文说明

普通说明文字建议：

- `font-sans`
- `leading-relaxed`
  或
- `leading-normal`

避免出现：

- 衬线体正文过密
- 说明文字行高过紧导致阅读发闷

---

## 七、字距与大小写规则

### 中文文本

- 默认不使用 `uppercase`
- 默认不使用过大的 `tracking`
- 常规中文正文不建议超过 `tracking-wide`

### 英文标签 / 角标

可在以下场景适度使用：

- `uppercase`
- `tracking-wide`
- `tracking-wider`

适用对象：

- 英文角标
- 极短状态标签
- 装饰性胶囊标签

禁止事项：

- 中文长文本禁止强制大字距
- 中文正文不要为了“高级感”随意拉开字间距

---

## 八、组件级字体使用建议

### Header / 顶部主标题

- 标题：`font-serif`
- 按钮与图标辅助信息：`font-sans`

### Home 首页

- 黄历/奇门标题：`font-serif`
- 说明、提示、按钮：`font-sans`
- 余额主数值：`font-serif`
- 按钮说明里的积分数字：跟随 `font-sans`
- 对齐敏感的数据列再使用 `font-mono`

### Analysis 评测页

- 术数盘面、宫位、门星神：`font-serif`
- 结果主分值、结果页手机号标题：`font-serif`
- 输入框、按钮、弹窗里的数字：跟随 `font-sans`
- 客服号、结构化联系信息：`font-mono`
- 表单、说明、按钮、弹窗：`font-sans`

### AIAgent 智能体页

- 页面主标题可保留 `font-serif`
- 聊天正文、输入框、按钮、快捷提问：统一 `font-sans`
- 时间、调试信息、编号：`font-mono`

### Profile 我的页

- 账户昵称标题：`font-sans`
- 积分主数值：`font-serif`
- 记录列表中的时间、差额等结构化字段：`font-mono`
- 普通数量徽标、评测号码正文：跟随 `font-sans`
- 卡片说明、操作按钮：`font-sans`

---

## 九、当前项目已识别的主要问题

当前前端已经出现以下字体治理问题：

1. 仅定义了 `sans / serif`，尚未建立 `mono` 语义层
2. 存在大量零散字号
3. `font-serif` 在部分位置承担了正文职责
4. `font-black` 使用较多，容易造成中文界面压迫感
5. 不同页面对数字字体的策略不一致，曾出现“所有数字都被等宽化”的过度设计

---

## 十、落地优先级

### 第一阶段：基础设施

- 在 `index.css` 中定义 `--font-sans`、`--font-serif`、`--font-mono`
- 增加对应语义工具类
- 为结构化数据补充等宽数字能力，而不是覆盖所有数字

### 第二阶段：高频页面

- `Header`
- `Home`
- `Analysis`
- `Profile`
- `AIAgent`

### 第三阶段：收敛细节

- 统一字号 token
- 减少非常规像素值
- 收敛 `font-black`
- 统一角标、标签、状态字的字距策略

---

## 十一、协作约束

后续新增 UI 时，默认遵循以下提交流程：

1. 先判断文本属于哪种语义：文化 / 说明 / 数据
2. 再决定使用 `font-serif`、`font-sans` 或 `font-mono`
3. 优先复用既有字号层级
4. 大字号衬线标题检查行高
5. 数据字段检查是否真的需要等宽与固定容器

如果后续需要，我建议继续补一份：

- 字体 token 命名规范
- 页面常用 typography utility 清单
- 组件示例对照表
